Stop. Do not redesign the whole Convention Builder again. We are now building the next layer of the same system: re-run discovery with new evidence on the same project, compare it against the current convention understanding, and show the delta before accepting changes.

Critical rule:
We do not use preview.
We do not use mock data.
We do not use a temporary or local database.
All work must run against the live production database only.

Do not touch Files, RFIs, Submittals, Dashboard, or unrelated modules.
Modify only the Convention Builder flow and the minimum backend/database support required.

This is the goal:
A project convention is not one-time setup. It must be revisitable, expandable, and evidence-driven over time.
Users must be able to come back later, add more real project evidence, re-run discovery, and see what changed before accepting anything.

Build this in exact order:

STEP 1 — Persist accepted convention findings as a baseline snapshot.

Right now the system analyzes and proposes. We now need to preserve the accepted state.

Add support for saving an accepted convention snapshot that includes at minimum:
project_id
convention_version
accepted disciplines
accepted systems
accepted document types
accepted extra fields
accepted field order
analysis summary
ambiguities
user-confirmed notes if any already exist
created_at
created_by_id if available

If the current naming_conventions table is not the right place to store a full accepted snapshot history, add a new table for convention versions/history. Name it something clear, for example:
naming_convention_versions

It must support multiple versions per project.
Do not overwrite prior accepted versions.

STEP 2 — Add a clear way to re-run discovery later on the same project.

In the Convention Builder for an existing project with an already accepted convention, add a visible action:

Add More Evidence and Re-run Discovery

This must not reset the project.
This must not wipe the current convention.
This must reopen the evidence intake flow using the same project.

The user must be able to:
paste more text
upload more PDFs / spreadsheets / screenshots / sample files
re-run analysis against the same project

STEP 3 — Re-run discovery must use both:
the current accepted convention baseline
and the new evidence

When the user re-runs discovery, the backend must receive:
the current accepted convention snapshot
the previously known structure if available
the new uploaded/pasted evidence
the original setup context if available

Claude must not analyze the new evidence in a vacuum.
It must compare the new evidence against the existing accepted convention.

STEP 4 — Add a new backend endpoint for re-run discovery comparison.

Add a dedicated endpoint, for example:
POST /api/v1/projects/:projectId/convention/reanalyze

This endpoint must:
load the current accepted convention baseline
combine it with the new evidence
ask Claude to compare the baseline against the new evidence
return structured JSON describing what changed

Prompt intent in substance:
You are re-evaluating an existing project naming convention using newly provided evidence.
Do not discard the current convention blindly.
Identify what is confirmed, what is newly discovered, what conflicts with the current convention, and what remains unresolved.
Return ONLY valid JSON.

Return structure:
{
"baselineVersion": 1,
"analysisSummary": "",
"confirmedItems": {
"disciplines": [],
"systems": [],
"documentTypes": [],
"extraFields": [],
"fieldOrder": []
},

"disciplines": [],

"fieldOrder": []
},

{
"category": "",

}
],
"stillUnresolved": [],
"recommendedActions": [],
"proposedNextVersionSummary": ""
}

If parsing fails, return a clean structured error and do not crash the builder.

STEP 5 — Build a new review screen after re-analysis.

Add a new step/screen called:

Convention Changes Review

Current accepted version
What was confirmed by the new evidence
What is newly suggested
What conflicts with the current convention
What remains unresolved
Recommended actions
Proposed next-version summary

This must not feel like a fresh setup.
It must feel like:
existing convention + new evidence + change review

The user must be able to:

keep the current version’s item

Do not silently change the convention.
