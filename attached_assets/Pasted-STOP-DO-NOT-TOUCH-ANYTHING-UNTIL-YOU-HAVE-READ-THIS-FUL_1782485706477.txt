This builds real server-side Trade+Floor sequence authority for lens_viewpoints, plus a
viewpoint lifecycle feature (Edit / Reassign / Void). Confirmed by investigation: no atomic
counter mechanism exists anywhere in this codebase today (every existing "next number" is
non-atomic count/max/scan), display_id values in production do not follow any parseable
trade/floor/sequence pattern (two incompatible formats exist, with duplicates), and the
plugin's sync payload carries no group identifier today. Build in this exact order.

PART 1 — SCHEMA (drizzle schema file AND the idempotent app.ts migration block, both places,
per existing project convention):

New dedicated counter table, since no atomic counter mechanism exists anywhere in this
codebase to reuse:
  lens_viewpoint_sequence_counters
  - id serial PK
  - project_id integer NOT NULL
  - trade text NOT NULL
  - floor text NOT NULL
  - current_seq integer NOT NULL DEFAULT 0
  - UNIQUE (project_id, trade, floor)

Add to lens_viewpoints:
  - trade_floor_seq integer, nullable
  - trade_floor_seq_correction integer, nullable (the "R" number — null normally, 1/2/3... when
    the platform had to correct what the plugin submitted; display logic shows "-R{n}" suffix
    on the platform only when this is non-null)
  - issue_group_id text, nullable
  - lifecycle_status text NOT NULL DEFAULT 'active' (allowed values: active, superseded, voided)
  - supersedes_id integer, nullable, self-referencing FK to lens_viewpoints.id

PART 2 — ATOMIC SEQUENCE ASSIGNMENT (the real fix to the confirmed gap)

Implement the counter as a single atomic round trip per viewpoint, the standard race-safe
pattern (no existing precedent in this codebase to copy — this IS the new precedent):
  INSERT INTO lens_viewpoint_sequence_counters (project_id, trade, floor, current_seq)
    VALUES ($1, $2, $3, 1)
    ON CONFLICT (project_id, trade, floor)
    DO UPDATE SET current_seq = lens_viewpoint_sequence_counters.current_seq + 1
    RETURNING current_seq;
This single statement both creates the counter row on first use AND atomically increments it
on every subsequent call - no separate read-then-write, no race window, works correctly under
concurrent sync requests for the same project+trade+floor.

In the lens-sync route (clash_reports.ts ~489-630), for each viewpoint being newly inserted
(not the existing dedup-skip path): call the above atomic increment for (project_id, trade,
floor) to get the real next number, store it in trade_floor_seq.
- If the plugin's submitted display_id implies a sequence number matching what was just
  atomically assigned: no correction needed, trade_floor_seq_correction stays null.
- If it does NOT match (collision - two machines, offline gap, etc.): trade_floor_seq is the
  REAL assigned number; set trade_floor_seq_correction to the next correction number for this
  specific row (1 the first time, 2 if corrected again, etc.) so the platform can display
  "{trade}-{floor}-{trade_floor_seq}-R{trade_floor_seq_correction}". The sync response back to
  the plugin must include the real assigned number so its local cache can be updated. The
  ALREADY-SAVED Navisworks viewpoint object and its visible name/comment are NEVER rewritten by
  this process - Navisworks remains a frozen historical record of what was shown at save time.
