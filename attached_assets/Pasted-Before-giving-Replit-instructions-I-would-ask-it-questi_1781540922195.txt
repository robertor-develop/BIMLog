Before giving Replit instructions, I would ask it questions that verify the architecture, the database wiring, and the generator behavior. The goal is to avoid improving PLATFORM.md based on assumptions.

Ask Replit these:

Confirm runtime database source

“Inspect the actual runtime database initialization path. Which environment variable does the running API server use to create the Postgres pool at runtime: PROD_DATABASE_URL, DATABASE_URL, PGHOST/PGDATABASE, or something else? Show the exact file and line where the connection string is selected.”

This is the most important question. Everything else depends on this being true.

Confirm Drizzle migration database source

“Inspect the Drizzle config and migration scripts. Which database URL does drizzle-kit use during schema generation, push, or migrations? Confirm whether it uses DATABASE_URL, PROD_DATABASE_URL, or another variable.”

This checks whether the DATABASE_URL = heliumdb only claim is accurate.

Confirm whether DATABASE_URL can be renamed or isolated

“Can we safely rename or isolate the migration-only DATABASE_URL to something like MIGRATION_DATABASE_URL, or does Replit/Drizzle require the variable to remain named DATABASE_URL?”

This matters because DATABASE_URL is a dangerous name if it points to an ephemeral database.

Confirm generated-file lifecycle

“Confirm when PLATFORM.md is generated. Is it generated on every api-server build only, on dev startup, or on deployment? Show the relevant package.json script and generator invocation.”

This avoids false expectations about when the file updates.

Confirm generator source of truth

“Does generate-platform-md.ts scan the actual filesystem and source files, or does it use manually curated arrays for routes, gotchas, and sections? Identify which sections are automatic and which are manually maintained.”

This matters because some parts may go stale if they are curated.

Confirm route prefix enforcement

“Where is the global /api/v1 prefix applied? Confirm whether all route files are mounted only through routes/index.ts or whether any routes bypass the global prefix.”

This verifies that the /api/v1 rule is universal.

Confirm redirect behavior

“Search all backend route files for res.redirect. Which redirects exist, and do all of them include the /api/v1 prefix where required?”

This turns the gotcha into an actionable check.

Confirm route ordering risk

“Search for parameterized routes like /:id, /:reportId, or similar. Are there literal subpaths currently declared after parameterized catch-alls? Identify any route files where ordering could cause shadowing.”

This is important because route shadowing is a silent bug.

Confirm schema migration duplication

“List all startup migration blocks in artifacts/api-server/src/app.ts. Do they cover every current Drizzle schema table and column that production requires, especially lens_viewpoints, lens_viewpoint_reports, living_brief, linked_items, and reporting tables?”

This checks the current risk of Drizzle schema and startup migrations diverging.

Confirm production data-loss prevention

“Does the app log the actual runtime database host at startup using PROD_DATABASE_URL, without exposing credentials? If not, where should this diagnostic banner be added?”

This would prevent another heliumdb/Neon confusion.

Confirm auth and super-admin rules

“Inspect authMiddleware, requireProjectMember, requirePermission, and isSuperAdminMiddleware. Is it accurate that JWT payload carries isSuperAdmin, project guards allow super-admin bypass, and isSuperAdminMiddleware re-checks users.is_super_admin from the database?”

This verifies the auth section before documenting it as fact.

Confirm Living Brief gates

“Inspect the Living Brief routes and schema. Is access controlled by both password validation and eligibility through users.is_super_admin or users.can_access_living_brief? Where is the default password BIMAI360 defined or hashed?”

This checks a security-sensitive section.

Confirm build requirements

“Confirm whether artifacts/bimlog still requires PORT=3000 pnpm build, and whether artifacts/api-server still bundles to dist/index.cjs via esbuild. Show the relevant scripts.”

This prevents outdated build instructions.

Confirm whether PLATFORM.md should include a commit hash

“Can the generator include the current git commit SHA, package versions, and workspace name at generation time? If yes, add fields like Generated from commit, Generated by script, and Workspace root.”

This makes the file auditable.

Confirm safe output policy

“Can the generator redact secrets and avoid printing full database URLs, tokens, passwords, or connection strings? Confirm that any database diagnostic only shows host/database name, never credentials.”

This is necessary before adding stronger database diagnostics.

The best instruction to Replit should not be “improve this file” yet. It should be:

“First audit and answer these questions with file paths and line references. Do not modify code yet.”

That way you get a factual architecture audit before asking it to rewrite the generator.
