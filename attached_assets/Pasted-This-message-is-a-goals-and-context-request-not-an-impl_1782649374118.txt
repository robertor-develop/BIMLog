CONTEXT: Tonight's earlier work already added real, dedicated database columns to
lens_viewpoints for lifecycle tracking - revisionNumber, supersedesId, lifecycleStatus,
issueGroupId. That data model is sound and is NOT being rebuilt. The problem is entirely in

data itself.

THE REAL GOALS:

1. Every lifecycle/chain attribute (Group membership, lifecycle status -
   active/superseded/voided, Revision number, ID format) needs its OWN real column in the
   on-screen table - not text badges squeezed inline next to the ID. The current inline-badge
   approach (e.g. "Fire Protection-B1-4-R1  [Superseded]  Group") is hard to scan and was
   never asked for in this form.

2. The user needs independent control over what's visible - separate toggles for: show/hide
   Group indicator, show/hide lifecycle status, show/hide Revision number, choice of ID format
   (raw displayId vs trade-floor-seq code). Each toggle independent of the others - a person
   should be able to see ONLY active items with NO group/revision noise if that's what they
   want, or see everything, their choice.

3. PDF/export should respect the same filtering logic as the on-screen view by default. A
   voided viewpoint cluttering a clean coordination report makes no sense as a default - the
   report a person prints should reflect what they're actually choosing to see, the same way
   the live table does.
